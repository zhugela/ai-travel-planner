## 一、AI 大模型概念

### 1.1 什么是 AI 大模型？

**大模型（LLM，Large Language Model）** 是在海量数据上预训练的深度学习模型，参数量通常达亿～千亿级。核心机制是：根据输入上下文 **预测下一个 token**，从而生成连贯文本。

对开发者而言，大模型是 **可调用的智能服务**（API / SDK），而不是自己从零训练模型。

### 1.2 AI 大模型的分类

#### ① 按模态分类

| 类型 | 输入 | 代表 |
|------|------|------|
| 文本模型 | 文字 | Qwen、GPT、DeepSeek |
| 多模态模型 | 文字 + 图/音/视频 | Qwen-VL、GPT-4o |
| 语音模型 | 音频 | Whisper、CosyVoice |
| 嵌入模型 | 文本 → 向量 | text-embedding-v3 |

#### ② 按开源性分类

| 类型 | 特点 | 例子 |
|------|------|------|
| 闭源商用 API | 按量付费、能力强 | GPT-4、通义千问 API |
| 开源可部署 | 可下载权重、私有化 | Llama、Qwen2.5、Gemma |

#### ③ 按规模分类

| 规模 | 参数量级 | 特点 |
|------|----------|------|
| 小模型 | < 10B | 本地可跑、速度快 |
| 中模型 | 10B～100B | 能力与成本平衡 |
| 大模型 | > 100B | 能力强、需云端或高端 GPU |

#### ④ 按用途分类

| 用途 | 说明 |
|------|------|
| 通用对话 | 问答、写作、编程 |
| 代码模型 | 专精代码（CodeQwen 等） |
| 推理模型 | 强化逻辑（DeepSeek-R1 等） |
| Embedding | 向量化，用于 RAG |
| Rerank | 检索结果精排 |

### 1.3 开发者怎么学习 AI 大模型？

```text
会用 → 会配 → 会搭 → 会优
 │      │      │      │
调API  Prompt  RAG    上线
       选型    Agent  评测
```

**建议**：以「能跑通 Demo → 能写进 Java 项目」为主线，不要一上来啃论文。本章的 `demo/invoke` 就是第一阶段。

### 1.4 如何对比和选择大模型？

| 维度 | 看什么 |
|------|--------|
| 能力 | 推理、中文、代码、多模态 |
| 成本 | 每百万 token 价格、免费额度 |
| 速度 | 首 token 延迟 |
| 合规 | 国内可用性、数据出境 |
| 生态 | SDK、Spring AI 支持、文档 |
| 部署 | 仅 API 还是可本地 Ollama |

**本项目选型**：

- **主线**：阿里云百炼 Qwen（国内稳定、文档全）
- **备用**：Ollama 本地模型（省钱、离线调试）

## 二、接入 AI 大模型

### 2.1 使用大模型的 2 种途径

| 途径 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **云服务** | 调用厂商 API（百炼、OpenAI 等） | 免运维、模型新、能力强 | 按量付费、数据上云 |
| **自部署** | Ollama、vLLM 等在本地/私有云跑开源模型 | 数据私有、长期可能更省 | 要硬件与运维 |

### 2.2 接入大模型的 3 种方式

| 方式 | 谁在用 | 典型场景 | 本项目 |
|------|--------|----------|--------|
| **① AI 应用平台** | 产品/运营 | 百炼控制台建智能体、工作流 | 课前准备，低代码验证 |
| **② AI 软件客户端** | 个人 | ChatGPT App、通义 App、Cursor | 体验能力，不写代码 |
| **③ 程序接入** | 开发者 | SDK / HTTP / 框架 | **本章重点** |

```text
平台接入 → 快速验证 Prompt 和工作流
程序接入 → 产品化，集成进 Spring Boot
```

### 2.3 百炼平台准备（课前必做）

在 [百炼控制台](https://bailian.console.aliyun.com/) 完成：

| 类型 | 操作 | 完成标志 |
|------|------|----------|
| 智能体应用 | 创建 → 配模型和提示词 → 试运行 → 发布 | 状态「已发布」 |
| 工作流应用 | 画布：开始 → 大模型 → 结束 → 发布 | 状态「已发布」 |
| 智能体编排 | 新版：工作流 + **智能体群组** 节点 | 状态「已发布」 |

> 2025 年 8 月起，「智能体编排」已并入工作流模块，用 **智能体群组** 节点代替原入口。

**账号要求**：实名 + 有余额（欠费报 `Arrearage` 错误码）。

## 三、后端项目初始化

### 3.1 环境准备

| 工具 | 要求 |
|------|------|
| JDK | 21 |
| Maven | 3.8+ |
| IDEA | 推荐 IntelliJ IDEA |
| 百炼账号 | API Key + 充值 |

**环境变量（必配）**：

```bash
DASHSCOPE_API_KEY=sk-你的密钥
```

IDEA 配置路径：Run Configuration → Environment variables → 添加上述变量。

### 3.2 新建 / 打开项目

| 项 | 值 |
|----|-----|
| 框架 | Spring Boot 3.4.4 |
| 包名 | `com.yupi.aitravelplanner` |
| 启动类 | `AiTravelPlannerApplication` |
| 端口 | `8123` |
| 上下文 | `/api` |

**验证启动**：

```text
GET http://localhost:8123/api/health  →  ok
```

### 3.3 整合依赖

#### Hutool 工具库

- **依赖**：`cn.hutool:hutool-all:5.8.37`
- **用途**：HTTP 请求、JSON 拼装
- **本项目**：`HttpAiInvoke` 使用 `HttpRequest`、`JSONObject`

#### Knife4j 接口文档

- **依赖**：`knife4j-openapi3-jakarta-spring-boot-starter:4.4.0`
- **访问**：`http://localhost:8123/api/doc.html`
- **配置**：`application.yml` 中 `knife4j`、`springdoc` 段

#### 本章核心 AI 依赖

| 依赖 | 作用 |
|------|------|
| `dashscope-sdk-java` | 官方 SDK |
| `spring-ai-alibaba-starter-dashscope` | Spring AI 接百炼 |
| `spring-ai-starter-model-ollama` | Spring AI 接 Ollama |
| `langchain4j-community-dashscope` | LangChain4j 接百炼 |

**版本注意**：Spring AI 统一用 BOM `1.0.0`，**不要**混用 `1.0.0-M6` 等里程碑包。

**SLF4J 注意**：`dashscope-sdk-java` 排除 `slf4j-simple`，避免与 Logback 冲突。

### 3.4 配置文件要点

文件：`src/main/resources/application.yml`

```yaml
spring:
  ai:
    dashscope:
      api-key: ${DASHSCOPE_API_KEY}   # 百炼密钥，走环境变量
    ollama:
      base-url: http://localhost:11434
      chat:
        options:
          model: gemma3:1b              # 本地 Ollama 模型名
```

## 四、程序调用 AI 大模型

本章项目提供 **5 种调用方式**，均在 `src/main/java/com/yupi/aitravelplanner/demo/invoke/` 下。

### 4.1 SDK 接入

**文件**：`SdkAiInvoke.java`

**核心流程**：

```text
Generation → 构建 Message（system + user）→ GenerationParam → gen.call()
```

**关键代码逻辑**：

```java
GenerationParam param = GenerationParam.builder()
    .apiKey(TestApiKey.API_KEY)   // 建议改为环境变量
    .model("qwen-plus")
    .messages(Arrays.asList(systemMsg, userMsg))
    .resultFormat(GenerationParam.ResultFormat.MESSAGE)
    .build();
return gen.call(param);
```

**运行方式**：IDEA 右键 `SdkAiInvoke.main()` 单独运行。

| 优点 | 缺点 |
|------|------|
| 功能最全、官方维护 | 绑定 DashScope，换厂商要改代码 |
| 错误信息清晰 | 需单独管理 API Key |

### 4.2 HTTP 接入

**文件**：`HttpAiInvoke.java`

**请求地址**：

```text
POST https://dashscope.aliyuncs.com/api/v1/services/aigc/text-generation/generation
Authorization: Bearer sk-xxx
Content-Type: application/json
```

**请求体结构**：

```json
{
  "model": "qwen-plus",
  "input": {
    "messages": [
      { "role": "system", "content": "..." },
      { "role": "user", "content": "..." }
    ]
  },
  "parameters": { "result_format": "message" }
}
```

**运行方式**：IDEA 右键 `HttpAiInvoke.main()` 单独运行。

| 优点 | 缺点 |
|------|------|
| 理解底层协议 | 手写 JSON 繁琐 |
| 任何语言都能学 | API 升级需自己跟文档 |
| 依赖最少 | 无 Spring 集成 |

### 4.3 Spring AI 接入（项目主线）

**文件**：`SpringAiAiInvoke.java`

**配置**：`spring.ai.dashscope.api-key`（见 `application.yml`）

**核心代码**：

```java
@Resource
private ChatModel dashscopeChatModel;

AssistantMessage msg = dashscopeChatModel
    .call(new Prompt("你好，我是鱼皮"))
    .getResult()
    .getOutput();
```

**运行方式**：

1. 配置 `DASHSCOPE_API_KEY` 环境变量
2. 取消类上的 `//@Component` 注释
3. 启动 `AiTravelPlannerApplication`，启动时自动执行

| 优点 | 缺点 |
|------|------|
| 统一 `ChatModel` 接口，换模型只改配置 | 依赖较多 |
| 与 Spring Boot 深度集成 | 版本需对齐 BOM |
| 后续 RAG、Advisor、MCP 都基于此 | — |

> **结论**：业务开发以 Spring AI 为主线。

### 4.4 LangChain4j 接入

**文件**：`LangChainAiInvoke.java`

**核心代码**：

```java
ChatLanguageModel model = QwenChatModel.builder()
    .apiKey(TestApiKey.API_KEY)
    .modelName("qwen-max")
    .build();
String answer = model.chat("...");
```

**运行方式**：IDEA 右键 `LangChainAiInvoke.main()` 单独运行。

| 优点 | 缺点 |
|------|------|
| Agent 链式编排能力强 | 另一套 API，学习成本略高 |
| 社区活跃 | 与 Spring AI 并存，用于对比 |

### 4.5 接入方式对比总结

| 方式 | 类 | 学习成本 | Spring 集成 | 推荐场景 |
|------|-----|----------|-------------|----------|
| SDK | `SdkAiInvoke` | 低 | 一般 | 快速验证、平台特有能力 |
| HTTP | `HttpAiInvoke` | 中 | 无 | 理解原理、排查问题 |
| Spring AI | `SpringAiAiInvoke` | 中 | **最好** | **生产主线** |
| LangChain4j | `LangChainAiInvoke` | 中高 | 可集成 | Agent 编排、框架对比 |
| Ollama | `OllamaAiInvoke` | 低 | 好（Spring AI） | 本地免费调试 |

**选用原则**：

```text
业务开发  → Spring AI
排查问题  → HTTP
验证平台  → SDK
本地实验  → Ollama
框架对比  → LangChain4j
```

## 五、扩展知识：本地部署与接入 Ollama

### 5.1 本地安装大模型

1. 下载安装 [Ollama](https://ollama.com/)
2. 拉取模型：

```bash
ollama pull gemma3:1b
# 或：ollama pull qwen2.5:3b
```

3. 确认服务运行：`http://localhost:11434`

### 5.2 Spring AI 调用 Ollama

**文件**：`OllamaAiInvoke.java`

**配置**（`application.yml`）：

```yaml
spring.ai.ollama.base-url: http://localhost:11434
spring.ai.ollama.chat.options.model: gemma3:1b
```

**核心代码**：

```java
@Resource
private ChatModel ollamaChatModel;  // Bean 名与百炼的 dashscopeChatModel 区分

ollamaChatModel.call(new Prompt("你好")).getResult().getOutput();
```

**与百炼调用的对比**：业务代码几乎相同，只换注入的 `ChatModel` Bean，体现 Spring AI 的 **模型无关性**。

**注意**：本地小模型能力弱于云端 `qwen-plus`，适合开发调试，复杂旅行规划仍建议用云端模型。

## 六、扩展思路

1. **多模型路由**：简单问答走 Ollama，复杂规划走 Qwen-Plus，按请求类型切换 `ChatModel`。
2. **流式输出**：使用 `StreamingChatModel`，提升对话体验（打字机效果）。
3. **统一异常处理**：封装 `Arrearage`（欠费）、限流、超时等为业务异常。
4. **Key 安全管理**：`TestApiKey.java` 仅用于本地测试，生产必须用环境变量，禁止提交 Git。
5. **从 Demo 到 API**：将 invoke 逻辑迁移到 `Controller` + `Service`，对外暴露 REST 接口。

## 八、踩坑记录

| 现象 | 原因 | 解决 |
|------|------|------|
| `Arrearage` | 百炼账号欠费 | 阿里云费用中心充值 |
| `ChatClientInputContentObservationFilter` 找不到 | Spring AI 混用 M6 与 1.0.0 | 删除 `spring-ai-ollama-spring-boot-starter` M6 依赖 |
| `DashScope API key must be set` | 未配置 Key | 设置环境变量 `DASHSCOPE_API_KEY` |
| `InvalidApiKey`（Key 已填仍失败） | Key 写在 Active profiles 而非 Environment variables | IDEA Run Configuration → Environment variables |
| `Failed to determine a suitable driver class` | 引入 JDBC 但未配数据源 | 暂不引入 PG/JDBC 依赖，或配置 PostgreSQL |
| `Port 8123 already in use` | 上次进程未退出 | 结束旧 Java 进程或修改 `server.port` |
| SLF4J multiple providers 警告 | dashscope-sdk 带入 slf4j-simple | pom 中排除 `slf4j-simple` |
| SDK 退出码 0 但 API 失败 | main 捕获异常后 `System.exit(0)` | 正常，看 stderr 错误信息即可 |
| Knife4j `ControllerAdviceBean` NoSuchMethodError | springdoc 2.3 与 Boot 3.4 不兼容 | knife4j 4.5 + springdoc 2.8.9（见第 03 章） |

> 每条踩坑的 5 Whys、反思日记、可复用规律见 [深度思考练习手册](./深度思考练习手册.md)。

### 8.1 踩坑 → 5 Whys 示例（Arrearage）

| 层 | 追问 | 答案 |
|----|------|------|
| 1 | 为什么 API 调不通？ | 返回 `Arrearage` |
| 2 | 为什么有 Key 还欠费？ | 百炼按账户余额计费，与「有 Key」无关 |
| 3 | 为什么 exit code 仍是 0？ | Demo 的 main 捕获异常后正常退出 |
| 4 | 为什么应先查账户再查代码？ | 外部 API 失败可能是商业状态而非逻辑 bug |
| 5 | **本质** | **云 API = 按量付费服务，账户与密钥是两类前置条件** |

### 8.2 规律栏（第 02 章提炼）

| 规律 | 说明 |
|------|------|
| 先外部后内部 | API 失败先查余额、Key、网络，再查代码 |
| BOM 统一版本 | Spring AI 系依赖只走 `spring-ai-bom`，不混 M6 |
| env ≠ profile | `DASHSCOPE_API_KEY` 是环境变量，不能写进 Active profiles |
| Demo 与生产分离 | `TestApiKey.java` 仅本地；禁止提交 Git |
