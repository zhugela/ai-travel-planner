# 一、什么是智能体?

## 1.1 定义

**智能体(Agent) = 以大语言模型为核心**,融合记忆 / 知识库 / 工具调用能力的 AI 系统。

**vs 普通 LLM 的 6 大区别**:

| # | 能力 | 普通 LLM | 智能体 |
|---|---|---|---|
| 1 | 感知环境 | ❌ | ✅ |
| 2 | 自主任务拆解 | ❌ | ✅ |
| 3 | 主动调用工具 | ❌ | ✅ |
| 4 | 多步链式推理 | ❌(单轮) | ✅(CoT) |
| 5 | 长期记忆 | ❌(单次) | ✅ |
| 6 | 反馈动态调优 | ❌ | ✅ |

**日常生活例子**:**DeepSeek「深度思考」** = 轻量化智能体能力 — 先分步推理,再出答案,不是直接给。

## 1.2 三层分类(自主性递增)

# 二、智能体实现关键技术

4 大底层支撑技术,是所有自主智能体的通用基础。

## 2.1 CoT 思维链(Chain of Thought)

**作用**:让 AI 模仿人类**分步思考**,处理复杂推理问题时,先拆解推导,再输出答案。

**2 种落地方式**:
1. **基础 Prompt 引导**:`请一步一步思考问题`
2. **Few-shot 少样本**:给模型看带完整思维过程的示例

**OpenManus 例子**:
```python
Thinking: 详细拆解推理过程
Answer: 最终精简答案
```

## 2.2 Agent Loop 执行循环(智能体最核心运行机制)

**定义**:**脱离用户实时输入**,智能体自主循环执行「推理→工具→结果」,直到任务完成或达最大步数。

**vs 传统 LLM**:
- 传统 LLM:一问一答,单次交互
- Agent Loop:单次输入后,AI 自主多轮工具调用,**无需用户再干预**

**Java 伪代码**:
```java
while(currentStep < MAX_STEPS && !isFinished()){
    currentStep++;
    String stepResult = executeStep();
}
```

## 2.3 ReAct 模式(Reasoning+Acting)— **工业界主流**

4 步核心循环:
1. **Reason 推理**:拆解任务,确定下一步动作
2. **Act 行动**:调用外部工具(浏览器 / 代码 / 搜索 / DB)
3. **Observe 观察**:捕获工具返回结果
4. **循环迭代**:回到 Step 1,直到任务完成

**Java 演示代码**:
```java
void executeReAct(String task) {
    String state = "开始";
    while (!state.equals("完成")) {
        String thought = "思考下一步行动";
        String action = "执行具体操作";
        String observation = "观察执行结果";
        state = "完成";
    }
}
```

## 2.4 4 大支撑系统

| 系统 | 作用 |
|---|---|
| **AI 大模型** | 智能体大脑 |
| **记忆系统** | 存对话历史 + 中间状态 |
| **知识库(RAG)** | 弥补静态知识局限 |
| **工具调用系统** | 跳出自有模型,访问外部能力 |

**特殊工具**:**Compute Use** = AI 直接操控本机(代码 / 文件 / 系统),⚠️ **必须在沙箱/虚拟机跑**,防止误删文件。

# 四、四层 Agent 核心源码逐段解析

## 4.1 BaseAgent 基础层(模板方法设计模式)

```python
# 主循环 while
while current_step < max_steps and state != FINISHED:
    current_step += 1
    step_result = await self.step()  # 子类实现
    if self.is_stuck():
        self.handle_stuck_state()  # 防死循环
```

**职责**:
- 管理 IDLE / RUNNING / FINISHED
- 读写记忆
- 限最大步数
- 抽象 `step()`,子类实现「思考+行动」

## 4.2 ReActAgent 第二层(落地 ReAct 思想)

```python
async def think(self) -> bool: ...   # 推理:要不要行动
async def act(self) -> str: ...      # 行动:调工具/输出
async def step(self):
    should_act = await self.think()
    if not should_act: return "思考完毕,无需操作"
    return await self.act()
```

**单步固定流程**:先推理,再行动。**不**实现工具调用,仅流程骨架。

## 4.3 ToolCallAgent 第三层(框架核心)

### think() 核心逻辑
1. 历史对话 + 下一步提示词 → `llm.ask_tool()` 传入所有工具
2. LLM 返回要调用的工具名 + 参数
3. 保存到 `self.tool_calls` + 记录文本内容
4. `return bool(self.tool_calls or content)` 告诉上层

### act() 核心逻辑
1. 无工具调用:返回最后一条文本,本轮结束
2. 遍历 `tool_calls` 批量执行:
   - `await self.execute_tool(command)`
   - 工具结果封装为 `tool_message` 存记忆(Observe)
3. 收集所有结果 → 回到 BaseAgent 循环 → 下一轮 think

**完整闭环**:`think(选工具) → act(执行 + 存结果) → BaseAgent 循环 → 再次 think`。

## 4.4 Manus 第四层(成品)

继承 ToolCallAgent,预注册:
- **本地工具集**:PythonExecute / BrowserUseTool / StrReplaceEditor / AskHuman / Terminate
- **MCP 客户端**:`mcp_clients: MCPClients`

重写 `think()` 做浏览器特殊适配(运行时补充网页上下文,不影响全局)。

# 五、MCP 协议深度集成(关键)

## 5.1 MCPClients:MCP 服务统一管理器

**继承 `ToolCollection`**,完全复用本地工具管理逻辑,实现**远程 MCP 工具与本地工具无差别调度**。

**核心成员**:
- `sessions`:存多个 MCP 服务会话,支持同时连多台远程 MCP
- `exit_stacks`:异步资源栈,自动管理 MCP 连接释放,避免连接泄漏
- 构造函数初始化空工具列表,统一对外命名 `mcp` 保证版本兼容

## 5.2 MCPClientTool:远程工具代理层

所有 MCP 服务暴露的远程工具都被封装为 `MCPClientTool`,**统一继承 `BaseTool`**,对上层 Agent 完全透明:

1. 存储会话 / 服务 ID / 原始远程工具名
2. `execute()` 逻辑:
   - 校验MCP 会话连接状态,未连接直接返回错误
   - 通过会话发起远程工具调用 `session.call_tool()`,传入参数
   - 解析服务返回的文本内容,封装为标准 `ToolResult` 返回
   - 全局捕获异常,统一包装工具执行错误

**设计价值**:**上层 Agent / LLM 不需要区分本地/远程工具**,调用逻辑完全一致,实现远程能力无缝扩展。

## 5.3 Manus 接入 MCP 流程

`connect_mcp_server()` 方法实现MCP 服务动态接入:

1. 区分两种连接模式:
   - `use_stdio=true`:本地子进程MCP 服务(stdio 通信)
   - `use_stdio=false`:网络 SSE 长连接远程 MCP 服务
2. 连接完成后,**自动提取该服务下全部远程工具实例**
3. **将远程工具批量加入** Manus 自身的 `available_tools` 工具集合
4. LLM 获取工具列表时,远程MCP 工具会**和本地 Python / 浏览器工具一同下发**,推理时可自由选择调用

# 六、开源项目可复用优质底层设计

## 6.1 PythonExecute 代码沙箱(安全执行 4 大最佳实践)

提供**隔离、可控**的 Python 代码运行环境:

1. **进程隔离**:`multiprocessing.Process` 新开独立进程,代码崩溃不会破坏主智能体进程
2. **超时熔断**:设执行超时,超时直接终止进程,防止无限死循环卡死智能体
3. **受限安全环境**:拷贝纯净内置函数,屏蔽文件删除 / 系统命令等高风险权限
4. **输出捕获**:重定向标准输出,完整收集代码打印结果,统一返回执行成功标识 + 观测文本

## 6.2 BaseAgent 状态上下文管理器 `state_context`

基于**异步上下文管理器**实现安全状态切换:

1. 进入上下文前记录原始状态,切换为目标运行状态
2. 代码块执行异常时,自动将智能体状态置为 `ERROR`
3. `finally` 块强制恢复原始状态,杜绝状态卡死 / 状态错乱
4. 适用场景:工具执行 / 大模型推理 / MCP 连接等易出错流程

## 6.3 ToolResult 统一工具结果模型

标准化所有工具(本地 / MCP 远程)的返回格式,**支持多工具结果拼接**:

- 字段:`output` 正常输出 / `error` 错误信息 / `base64_image` 多媒体 / `system` 系统日志
- 重载 `__add__` 运算符,支持批量工具执行后**自动合并输出 / 错误 / 图片内容**
- 统一结果格式大幅简化记忆存储、结果回传给 LLM 的逻辑

# 七、Java 自研简化版 YuManus(实战,课程标准方案)

## 7.1 数据模型定义

```java
public enum AgentState {
    IDLE,     // 空闲,可接收新任务
    RUNNING,  // 循环执行任务中
    FINISHED, // 任务正常完成
    ERROR     // 执行异常终止
}
```

**作用**:管控智能体生命周期,防止重复启动、异常状态未重置。

## 7.2 BaseAgent 关键设计 3 点

1. **解耦大模型**:`chatClient` 由外部传入,不硬编码模型,支持切换通义 / Claude / OpenAI 等任意 LLM
2. **对话记忆容器**:`messageList` 手动维护完整消息上下文,复刻 OpenManus 的 Memory 逻辑
3. **状态机控制**:`state` 字段搭配 `AgentState` 枚举,约束智能体只能在 `IDLE` 时启动,异常/完成自动切换状态

**核心抽象方法**:
- `abstract String step()`:单步执行逻辑(模板方法模式)
- `protected void cleanup()`:空实现,子类可重写释放 MCP 连接、文件句柄等资源

## 7.3 ReActAgent 拆分

```java
abstract class ReActAgent extends BaseAgent {
    abstract boolean think();  // 推理:分析上下文、选择工具
    abstract String act();     // 行动:调工具、获取观测

    @Override
    String step() {
        try {
            boolean shouldAct = think();
            if (!shouldAct) return "思考完成 - 无需行动";
            return act();
        } catch (Exception e) {
            e.printStackTrace();
            return "步骤执行失败:" + e.getMessage();
        }
    }
}
```

**单步固定流程**:先思考 → 判断 → 行动;**统一捕获单步异常**,保证循环不崩溃。

## 7.4 act 完善:检测 TerminateTool 改 state

```java
// 提取本轮所有工具返回结果
ToolResponseMessage toolResponseMessage = CollUtil.getLast(...);
// 判断本次是否调用了 TerminateTool 的 doTerminate 方法
boolean terminateToolCalled = toolResponseMessage.getResponses().stream()
    .anyMatch(response -> "doTerminate".equals(response.name()));
if (terminateToolCalled) {
    setState(AgentState.FINISHED);
}
```

**作用**:识别 LLM 主动调用结束工具 → 改 `state=FINISHED` → BaseAgent 主循环检测 → 跳出循环。
**防死循环**核心:防止 AI 反复执行工具,**自主判断完工、主动停机**。

## 7.5 ToolCallAgent 三种实现方案对比

| 方案 | 实现方式 | 适用场景 |
|---|---|---|
| **方案 1(本课采用)** | 关闭 Spring AI 自动工具托管,**手动**解析工具调用 + 执行 + 回填 | 学习 ReAct 完整闭环,流程透明可控 |
| 方案 2 | 完全依赖 Spring AI 内置工具调用,封装极简 | 业务快速开发,不需要理解底层 |
| 方案 3 | 原生手写 Prompt 引导模型输出工具 JSON,无框架工具 | 学习底层原理,开发成本极高 |

### 类成员 + 构造器

```java
public class ToolCallAgent extends ReActAgent {
    private final ToolCallback[] availableTools;          // 全部可用工具
    private ChatResponse toolCallChatResponse;            // 大模型返回的工具调用指令
    private final ToolCallingManager toolCallingManager;  // 工具执行管理器
    private final ChatOptions chatOptions;                // 模型参数,关闭自动工具托管

    public ToolCallAgent(ToolCallback[] availableTools) {
        super();
        this.availableTools = availableTools;
        this.toolCallingManager = ToolCallingManager.builder().build();
        // 关键:proxyToolCalls=true = 代理工具调用,由业务代码手动处理
        this.chatOptions = DashScopeChatOptions.builder()
                .withProxyToolCalls(true)
                .build();
    }
}
```

⚠️ **阿里云 DashScope 适配注意**:
- ❌ **不能用** `internalToolExecutionEnabled=false`(会失效)
- ✅ **必须用** `withProxyToolCalls(true)`

### `think()` 推理方法

1. 追加下一步提示词到记忆列表
2. 携带全部工具 + 系统提示词调 LLM,获取返回
3. 提取模型输出文本与工具调用列表,打印日志
4. 分支:
   - 无工具调用 → 助手消息存记忆 → 返回 `false`(无需行动)
   - 有工具调用 → **不**存助手消息 → 返回 `true`(进入 act)
5. 全局异常捕获,异常消息写入上下文,返回 `false` 终止本轮

### `act()` 行动方法

1. 校验是否存在工具调用,无则直接返回
2. 调用 `toolCallingManager.executeToolCalls()` 批量执行工具
3. **自动生成新对话上下文**(包含助手消息 + 工具返回结果),**替换**原有 `messageList`
4. 遍历工具返回结果,拼接格式化字符串作为本轮输出

**底层原理**:`executeToolCalls` 内部自动拼接 `AssistantMessage` + `ToolResponseMessage`,**无需手动 add 消息**,避免上下文重复错乱。

## 7.5 TerminateTool(闭环关键,防无限循环)

### 工具定义

```java
@Tool(description = """
Terminate the interaction when the request is met OR if the assistant cannot proceed further with the task.
When you have finished all the tasks, call this tool to end the work.
""")
public class TerminateTool {
    public String doTerminate() {
        return "任务结束";
    }
}
```

### 接入流程

1. 在工具注册类 `ToolRegistration` 中,`TerminateTool` 与文件 / 搜索 / 浏览器等一起批量注册
2. 修改 `act()`:执行工具后**检测是否调用了 Terminate 工具**,若命中则将智能体 `state` 置为 `FINISHED`
3. 上层 `BaseAgent.run()` 主循环检测 `state == FINISHED`,**立刻跳出循环**,结束任务

## 7.6 整体四层数据流闭环

```
用户输入
  ↓
BaseAgent.run() 主循环
  ↓
ReActAgent.step()
  ↓
  1. think():LLM 选工具
  2. act():执行工具 + 回填观测
  ↓
循环重复,直到两种退出条件:
  (1) 达到 maxSteps 最大步数限制
  (2) LLM 调用 Terminate,state = FINISHED
  ↓
循环结束,拼接结果返回用户
```

## 9.2 MVP 结构

```java
@Component
public class SimpleAgent {
    @Resource private ChatClient chatClient;
    @Resource private ToolCallback[] tools;
    private final List<Message> history = new ArrayList<>();
    private static final int MAX_STEPS = 10;

    public String run(String userTask) {
        history.add(new UserMessage(userTask));
        for (int step = 0; step < MAX_STEPS; step++) {
            String response = chatClient.prompt()
                .messages(history)
                .tools(tools)
                .call()
                .content();
            history.add(new AssistantMessage(response));
            if (response.contains("DONE") || response.contains("任务完成")) {
                return response;
            }
        }
        return "达到最大步数";
    }
}
```

**这个 MVP** = Ch 06 @Tool + Ch 07 MCP + Ch 08 Agent Loop,合 3 章精华。

# 十、扩展:多智能体工作流 5 大模式

**单一 YuManus 只适合中小型任务,超复杂业务需要多智能体协同编排。**

## 10.1 Prompt Chaining(提示链)

- **逻辑**:任务拆成固定有序子步骤,上一步输出作为下一步输入,中间可加校验网关 Gate
- **示例**:写文章 → 生成大纲 → 撰写正文 → 润色校对;网关校验大纲不合格直接退出
- **适用**:流程固定、阶段清晰的标准化任务

## 10.2 Routing(路由分流)

- **逻辑**:路由智能体判断输入类型,分发至对应专业子智能体
- **示例**:客服系统 → 路由识别退款 / 咨询 / 故障 → 分流给退款 Agent / 问答 Agent / 运维 Agent
- **适用**:多类型输入、不同处理逻辑的分流场景

## 10.3 Parallelization(并行聚合)

- **逻辑**:同一任务拆分多路子任务并行执行,最后聚合器汇总全部输出
- **示例**:代码审计 → 3 个 Agent 同时审查同一段代码 → 聚合器汇总三方风险点投票
- **优势**:提速、多视角校验提升结果准确度

## 10.4 Orchestrator-Workers(协调者-工人)

- **逻辑**:一个总指挥 Orchestrator 拆解复杂任务,分发多个专业 Worker 智能体执行,最后合成器汇总
- **适用**:超大复杂无固定流程任务(市场调研、大型方案撰写),动态拆分

## 10.5 Evaluator-Optimizer(生成-评估迭代)

- **逻辑**:生成 Agent 产出初稿 → 评估 Agent 打分 + 修改建议 → 循环迭代直到达标;不达标则把反馈回传给生成器重写
- **示例**:翻译 / 文案优化 / 代码迭代 / 论文润色
- **核心**:闭环反馈迭代,持续优化输出质量

## 10.6 智能体编排框架

| 框架 | 语言 | 特点 |
|---|---|---|
| **LangGraph** | Python | LangChain 官方,用**状态图**描述智能体流程,支持分支/并行/循环/反思纠错,**实现 5 大模式全部** |
| **Spring AI Alibaba Graph** | Java | Spring AI 生态,封装多智能体协同 |
| **LiteFlow** | Java | 轻量级规则编排 |
| **Activiti** | Java | 老牌工作流引擎,集成 AI |

**核心代码模式**:
```java
StateGraph stateGraph = new StateGraph();
stateGraph.addNode("agent_a", agentA);
stateGraph.addNode("agent_b", agentB);
stateGraph.addEdge("agent_a", "aggregator");
stateGraph.addEdge("agent_b", "aggregator");
stateGraph.compile().execute(initialState);
```

**通过 `StateGraph` 定义全局状态**(消息列表 / 任务 / 上下文),注册多个智能体节点,配置节点流转条件,自动完成多 Agent 调度。

# 十一、扩展优化 4 大思路(生产级增强)

## 11.1 循环卡死检测机制(防无限死循环)

**核心思想**:**主动**干预循环,而不是被动等 `maxSteps` 兜底。

**代码逻辑**:
1. **阈值配置** `duplicateThreshold = 2`:连续 2 次出现高度相似的思考 / 工具调用,判定为卡死循环
2. **`isStuck()` 检测方法**:读取历史对话记忆,对比最近多轮 AI 输出内容,判断是否重复走完全一样的路径
3. **`handleStuckState()` 卡死处理**:
   - 向 `nextStepPrompt` 追加提示词,强制告知模型「不要重复无效操作,换全新策略」
   - 打印日志记录卡死事件,引导大模型跳出重复逻辑

**生活例子**:
你做约会攻略,连续 3 轮反复搜同一家商场、重复下载同一张图片:
1. 程序检测到重复行为 → 判定卡死
2. 强制给大脑加提醒:"别再搜这家店了,换别的商圈找方案"
3. 下一轮思考主动更换思路,避免无限循环消耗 token

**解决痛点**:仅靠 `maxSteps` 是**被动**兜底,循环检测是**主动**干预,提前跳出无效循环,节省 token + 提升执行效率。

## 11.2 人机交互 AskHuman(2 种实现方案)

### 方案 A:标准工具实现(Python 推荐工程化)

继承 `BaseTool` 标准化工具:
- 工具名 `ask_human`,入参 `inquire` 存放向用户提问的内容
- LLM 遇到信息不足时,主动调用该工具,框架统一接管输入输出
- **优势**:完全贴合整套工具调用体系,MCP / 本地工具统一管理,记忆自动记录交互记录

### 方案 B:标记文本解析(Java 简易版)

不新增工具,靠 Prompt 约定标记 `[ASK_USER]`:
1. 约束大模型缺信息时输出 `[ASK_USER]需要提问的内容`
2. 在 `think()` 方法中读取助手消息,检测标记
3. 截取问题,控制台弹出输入框接收人类回答,把回答追加进对话记忆,继续循环

**生活例子**:
做攻略时不知道另一半是否吃辣:
- **方案 A**:AI 调用「询问人类」工具,发起提问
- **方案 B**:AI 输出 `[ASK_USER]对方是否忌口辛辣食物?`

**两种方案对比**:

| 方案 | 优点 | 缺点 |
|---|---|---|
| 工具类 AskHuman | 标准化 / 兼容全部工具框架 / 可被 MCP 远程调用 | 需新增工具类,少量代码改动 |
| 文本标记解析 | 改动极小,不用新增工具 | 纯文本匹配易出错,无法适配 MCP 远程场景 |

## 11.3 接入 MCP 远程工具

基于已写完的 `ToolCallAgent` 复用逻辑,**无需重写**工具调度 / think / act 循环:

1. 连接 MCP 服务,拉取服务端暴露的全部远程工具
2. **将远程工具封装为和本地工具完全一致的工具列表**
3. 直接传入 `ToolCallAgent` 构造器,本地 / 远程工具对上层 AI **完全透明**

**生活例子**:
本地只有浏览器 / PDF 工具,想调用云端代码执行 / 数据库查询:
通过 MCP 对接远程服务,远程工具自动混入可用工具池,AI 自由选择本地或云端工具,**不用修改智能体核心逻辑**。

## 11.4 领域定制智能体 + 规划工作流

### 垂直领域专用智能体

直接继承 `ToolCallAgent`,**替换专属工具集 + 行业提示词**,快速定制专属智能体:
- **财务智能体**:内置报表 / 对账 / 税务工具
- **爬虫数据智能体**:内置网页抓取 / 清洗 / 存储工具

### 规划-执行工作流

参考 OpenManus `planning.py` 源码,新增分层规划能力:
1. 第一轮先拆分整体任务,生成分步执行计划
2. 后续循环**严格按照规划分步执行**,支持中途调整计划
3. 适合**长周期 / 复杂大型任务**(调研报告、全流程项目自动化)

## 11.5 整体拓展价值

5 个扩展点是**自研单智能体的生产级增强方案**:

| 维度 | 增强 | 扩展点 |
|---|---|---|
| **稳定性** | 杜绝无限循环 | 循环检测 + 终止工具双重机制 |
| **可用性** | 信息缺失时推进 | 人机交互 AskHuman |
| **能力** | 打通云端 | MCP 远程工具 |
| **业务定制** | 行业专属 | 领域定制智能体 |
| **复杂任务** | 长周期支持 | 规划-执行工作流 |
