# 一、整体概念:什么是 AI 服务化

## 1.1 定义

**AI 服务化 = 把本地跑的 AI 能力封装成可远程调用的 HTTP 接口**。
原本 AI 只能在 IDEA 里 `main()` 跑;服务化后,前端 / 第三方 / 手机 App 都能调。

## 1.2 核心技术:SSE(Server-Sent Events 服务端推送)

**核心痛点**:
- 同步接口 = 等 AI 生成完全部文字才一次性返回 → 用户盯着空白屏幕 10~30 秒
- SSE = **服务端边生成边往客户端推** → 打字机效果

**SSE 一句话原理**:
- HTTP 长连接 + `text/event-stream` MIME + `data:` 帧格式
- **单向**(服务端 → 客户端),HTTP 协议栈内搞定
- 浏览器用 `EventSource` 自动重连,服务端不断 `write` 就行
- **不是 WebSocket**(双向,独立协议,反代麻烦);SSE 是 HTTP,反代友好

## 1.3 整套流程分 4 大块

| # | 模块 | 本项目对应 |
|---|---|---|
| 1 | AI 对话应用流式/同步接口(普通聊天) | 「旅行规划问答」接口 |
| 2 | AI 智能体流式接口(多步工具调用) | `YuTravelAgent.runStream()` |
| 3 | Cursor AI 一键生成 Vue3 前端 | 「旅行规划」聊天页 + 「智能体」日志页 |
| 4 | Docker + Serverless 部署 | 微信云托管 / 阿里云 Serverless |

# 二、AI 应用接口开发(普通聊天,本项目对应「旅行问答」)

## 2.1 同步 vs 流式 选型

| 场景 | 用同步 | 用 SSE |
|---|---|---|
| 短问答 / 一两句话 | ✅ | 浪费 |
| 长文本生成 / 旅行攻略 | 体验差 | ✅ |
| Agent 多步推理 | ❌ 会卡死 | ✅ 必须 |

## 2.2 3 种 SSE 实现方案(核心对比)

| 方案 | 写法 | 适用栈 | 本项目选 |
|---|---|---|---|
| **方案 1** | `Flux<String>` + `produces` | **WebFlux 响应式** | ❌ |
| **方案 2** | `Flux<ServerSentEvent<String>>` | WebFlux,想控制 eventId | ❌ |
| **方案 3** | **`SseEmitter`**(Servlet 异步) | **传统 Spring MVC** | ✅ **主选** |

**为什么本项目主用方案 3**:
- 你目前的栈是 **Spring Boot 3 + Spring MVC**(非 WebFlux),[学习笔记-大纲篇.md:174](学习笔记-大纲篇.md) 明确写 `spring-boot-starter-web`
- SseEmitter 基于 Servlet 3.0 异步,**兼容老 Spring 项目**
- 手动管理:连接超时 + 消息推送 + 异常/结束回调 → 完全可控

## 2.3 底层流式调用方法(Service 层思路)

**关键点**:不要直接返回 `ChatResponse`(带 tool_calls / metadata 冗余),只 `.stream().content()` 返回纯文本 → **减少传输体积,提升流式效率**。

调用链:
```
ChatClient.prompt().user(msg).stream().content() → Flux<String>
                                                            ↓
                                                    .subscribe(
                                                        chunk -> emitter.send(chunk),    // 分片推
                                                        err   -> emitter.completeWithError(err),
                                                        ()    -> emitter.complete()
                                                    )
```

## 2.4 同步接口(短问答场景)

普通 `@GetMapping` 返回 `String` → 一次性拿完整回复。

## 2.5 测试 3 种方式

| 方式 | 优点 | 适用 |
|---|---|---|
| **curl 命令** | 终端实时看 `data:xxx` 分片 | ✅ **首选**,流式效果最直观 |
| 浏览器 F12 复制 curl | 网络面板右键一键复制 | 排查问题时 |
| IDEA HTTP Client | 图形化测试 | 非流式接口 |

# 三、AI 智能体接口开发(本项目对应 `YuTravelAgent`)

## 3.1 为什么 Agent 必须用 SSE

**普通 AI** = 1 步返回文字
**Agent** = **N 步循环**(think → act → tool → think...),每步 2~10 秒

| 方式 | 用户体验 | 服务端资源 |
|---|---|---|
| 同步 | 等几分钟看 1 次结果 | Tomcat 线程占满几分钟 |
| SSE | **每步立刻推一行日志** | 线程立刻释放 |

## 3.2 底层原理:为什么必须异步 + SSE

**致命约束**:**不能让 Agent 跑在 Tomcat 线程上**。
- Tomcat 默认 200 线程
- 每个 Agent 请求占满 1 个线程几分钟
- 10 个并发用户 → 线程池耗尽 → 整个网站挂掉

**正确做法**:`CompletableFuture.runAsync` 丢到独立线程池

```
请求进 Controller
  → new SseEmitter(5min)         // 1. 创建 SSE 句柄
  → CompletableFuture.runAsync(  // 2. 丢到后台线程池(不占 Tomcat)
        while 循环 step()
          每步 emitter.send("Step x: " + result)   // 3. 异步推
        emitter.complete()
    )
  → return emitter               // 4. Tomcat 线程立刻释放,只持有句柄
```

**本质**:Tomcat 线程只负责「接句柄 + 立刻返回」,推理在后台跑。

## 3.3 本项目 Agent 接入路径

```
Controller.doChatWithAgent(msg)
  → new YuTravelAgent(chatModel, allToolCallbacks)   // 每次新建,隔离 messageList
  → agent.runStream(msg)                              // 返回 SseEmitter
  → 前端 EventSource 监听,每步推一行日志
```

**⚠️ 复用 Ch 08 已实现的 4 层架构**:
- [BaseAgent.java](../src/main/java/com/yupi/aitravelplanner/agent/BaseAgent.java) → 加 `runStream()` 方法
- [YuTravelAgent.java](../src/main/java/com/yupi/aitravelplanner/agent/YuTravelAgent.java) → 直接 new,不用改
- `toolCallbacks` = Ch 06 的 7 个 `@Tool` + Ch 07 MCP 远程工具 → 直接注入

## 3.4 CORS 跨域配置(前后端不同端口必踩)

```java
// 全局 CORS 配置思路
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
            .allowedOriginPatterns("*")   // ⚠️ 不能用 allowedOrigins("*")
            .allowCredentials(true)
            .allowedMethods("GET","POST","PUT","DELETE","OPTIONS")
            .allowedHeaders("*");
    }
}
```

**⚠️ 致命坑**:`allowedOrigins("*")` 和 `allowCredentials(true)` **不能同时使用** → 触发 Spring 安全冲突 → 启动报错。
**改用**:`allowedOriginPatterns("*")` 兼容凭据。

## 3.5 Agent 接口测试(独立小节)

**测试方法与 §2.5 一致**:curl / F12 复制 / IDEA HTTP Client 三选一。但**观察点不同** —— Agent 接口不是看打字机,是看**步骤日志**。

**curl 命令样例**:
```bash
curl 'http://localhost:8123/api/agent/sse?message=帮我规划3天杭州旅行'
```

**预期输出**(终端持续打印):
```
data:Step 0 : ====第1步====
data:思考完成,准备调用工具: getAttraction
data:【工具执行】getAttraction => [杭州景点清单...]
data:Step 1 : ====第2步====
data:【工具执行】getWeather => [杭州未来7天天气...]
data:Step 2 : ====第3步====
...
data:[LoopGuard] 检测到死循环(重复工具或无进展),强制终止
```

**F12 网络面板观察 3 个点**:

| 观察项 | 正常 | 异常 |
|---|---|---|
| 响应头 `Content-Type` | `text/event-stream` | 别的 = 不是 SSE |
| 状态码 | **200** | 4xx/5xx = 后端崩了 |
| 帧格式 | 持续收到 `data:xxx` 帧,无固定结尾 | 一大坨 = 没流式 = Nginx buffering 没关 |
| 步骤数 | 等于 `YuTravelAgent.maxSteps`(默认 20) | 超 20 = 死循环,LoopGuard 没生效 |

**常见测试问题**:

| 现象 | 原因 | 解决 |
|---|---|---|
| 步骤数 = 1 就停了 | Agent 推理太快 / 提示词没要求工具 | 改 `systemPrompt` 强制要求工具 |
| 步骤数 = 20 卡死 | 模型陷死循环 | `LoopGuard` 已自动终止,看 `[LoopGuard]` 日志 |
| 工具返回 null | 工具注入失败,`toolCallbacks` 为空数组 | 检查 Controller 注入链路 |

# 五、AI 服务 Serverless 部署

## 5.1 基础概念

**Serverless = 不买云服务器,按调用次数 + 运行时长计费,闲置自动缩容到 0**。

| 维度 | 传统云服务器 | Serverless |
|---|---|---|
| 计费 | 24h/月 | 按请求+运行时长 |
| 闲置 | 扣费 | **不扣费**(副本 = 0) |
| 运维 | 自己装系统/拉镜像 | 平台托管 |
| 适合 | 长期稳定流量 | **流量波动大 / 演示项目 / 个人作品** |

## 5.2 主流平台

| 平台 | 适合 | 备注 |
|---|---|---|
| **微信云托管** | 鱼皮教程示例 | 微信生态友好 |
| 腾讯云 Serverless | 国内稳定 | 微信生态外可选 |
| 阿里云 Serverless | 国内稳定 | 阿里生态 |
| Railway | 海外/个人 | 简单 |
| **Vercel** | **前端专用** | 不用 Docker,一键部署 Vue |

## 5.3 Docker 是什么(一句话)

**应用 + 运行时 + 依赖 + 配置**全打进一个镜像 → 任意机器 `docker run` 跑出**完全一致**的环境。
镜像分层:改 1 行代码 → 只重新打包变更层 → 推送快。

## 5.4 后端部署(3 步)

### 步骤 1:`application-prod.yml` 生产配置
- 复制本地 `application.yml`
- **关闭开发日志**(避免线上日志爆炸)
- 配置线上大模型密钥(环境变量,别硬编码)
- 调整 SSE 长连接超时
- 启动时 `--spring.profiles.active=prod` 激活

### 步骤 2:Dockerfile(两种方案二选一)

| 方案 | 构建时机 | 镜像大小 | 适合 |
|---|---|---|---|
| **云端构建** | 平台拉源码 + Maven 打包 | 大(含源码+maven) | 自动化首选 |
| **本地预打 Jar** | 本地 `mvn package` → 只传 jar | 小 | 云端构建超时 |

**核心字段**(无论哪种方案):
- `EXPOSE 8123`(平台端口必须等于这个)
- `CMD ["java","-jar","...","--spring.profiles.active=prod"]`

**补充:Maven Wrapper**(`.mvn/` + `mvnw` 脚本):环境无 Maven 时自动下载对应版本,**保证构建一致性**。

### 步骤 3:微信云托管平台流程

1. 创建环境 → 新建后端服务
2. 上传代码 zip → 配置端口 `8123` → 勾选「存在 Dockerfile」
3. 发布 → 平台自动构建 + 启动
4. **排错关键**:先看「运行日志」,分清是**构建阶段报错**(Dockerfile / Maven)还是**启动阶段报错**(Java 异常 / 配置缺)
5. 成功 → 平台分配公网域名 → Swagger 在线测 SSE
6. 可绑定自定义域名

## 5.5 前端部署(Nginx + Docker)

### 5.5.1 部署规划

**为什么前端要 Nginx 反代**:
- 前后端不同域名 → 天然跨域
- **Nginx 反向代理 `/api` → 后端域名** → 浏览器只看到同源,**彻底消除 CORS**

### 5.5.2 前端生产环境适配

`src/api/index.js`:
```js
const API_BASE_URL = process.env.NODE_ENV === 'production'
  ? '/api'                       // 生产:相对路径,交 Nginx 转发
  : 'http://localhost:8123/api'  // 开发:直连后端
```

`NODE_ENV` 是 Vite 打包时自动注入,**不用手动配**。

### 5.5.3 nginx.conf 三大核心

| 配置块 | 作用 | 关键点 |
|---|---|---|
| `location / { try_files ... /index.html; }` | **Vue history 路由 404 兜底** | 必须有,否则刷新页面 404 |
| `location ~ /api/ { proxy_pass ...; }` | **API 反向代理** | 消除 CORS |
| SSE 专属:`proxy_buffering off; proxy_http_version 1.1; Connection "";` | **SSE 长连接不断开** | 不配 = 流式失效 |

**静态资源缓存**:`location ~* \.(js|css|png|jpg)$ { expires 1y; }` → 一年缓存。

### 5.5.4 前端 Dockerfile(多阶段构建)

**两阶段思路**(关键):
- **阶段 1**:`node:20-alpine` → `npm install && npm run build` 生成 `dist/`
- **阶段 2**:`nginx:alpine` → 复制 `dist/` + 自定义 `nginx.conf`
- **优点**:最终镜像里**没有 `node_modules`**(几百 MB),体积小到 50MB 左右

### 5.5.5 `.dockerignore`

打包时忽略:
- `node_modules` / `.idea` / `.vscode` / 本地 env
- npm 缓存 / logs / coverage
- 本地 `dist` / `build`

### 5.5.6 前端部署流程

1. 微信云托管新建前端服务,服务名 `yu-ai-agent-frontend`
2. **必须剔除 `node_modules`** 后打包 zip
3. 端口填 `80`(Nginx 默认 + Dockerfile `EXPOSE` + 平台配置 = 三者一致)
4. 勾选「存在 Dockerfile」→ 发布
5. 成功后前端公网域名可访问,EventStream 长连接持续返回

**备选**:Vercel / 腾讯云 Web 应用托管 → **前端专用,不用 Docker,一键部署 Vue**。

## 5.6 自动缩容省钱配置(核心)

**后端服务设置页面关键字段**:

# 九、踩坑预警(本章最容易踩的 5 个)

| # | 现象 | 原因 | 解决 |
|---|---|---|---|
| 1 | 部署后 SSE 没打字机 | **Nginx 默认 `proxy_buffering on`** 把流式缓冲成一次性 | `proxy_buffering off; proxy_http_version 1.1; Connection ""` |
| 2 | `allowedOrigins("*")` 启动报错 | Spring 安全:凭据 + 通配不能并存 | 改 `allowedOriginPatterns("*")` |
| 3 | 前端 SSE 一次性渲染 | 代码没拼接 EventStream 数据 | 丢给 Cursor 自动修 |
| 4 | Node `crypto.getRandomValues` 报错 | Node v16 < Vite 4.x 要求 | 升级 Node ≥18 |
| 5 | 上传 zip 几 GB / 构建超时 | 忘删 `node_modules` | `.dockerignore` + 手动剔除 |

# 十、本章小结

| 要点 | 一句话 |
|---|---|
| SSE 是什么 | HTTP 长连接 + `text/event-stream` + `data:` 帧,单向推送 |
| 为什么 Agent 必须异步 + SSE | 线程模型:不让 Agent 占 Tomcat 线程几分钟 |
| 三种 SSE 方案选哪个 | **本项目主用 `SseEmitter`**(Spring MVC 栈) |
| Cursor 4 要素 Prompt | 页面需求 + 技术栈 + 后端接口 + 系统适配 |
| Docker 多阶段构建 | Node 打包 + Nginx 跑静态,镜像瘦身 |
| Nginx SSE 关键 | `proxy_buffering off; Connection ""` |
| Serverless 省钱 | 实例副本最小值 = 0,闲置 0 费用 |
