## 一、决策四步法（通用框架）

每做一个技术决定，用下面四格填空（可抄进每日笔记）：

| 步骤 | 要问的问题 | 填写示例 |
|------|------------|----------|
| **为什么** | 要解决什么业务/技术问题？ | 旅行规划需要多轮澄清，模型必须记住前几轮 |
| **还有什么方案** | 至少列出 2 个替代方案 | 手写 get/add；PromptChatMemoryAdvisor；无记忆每轮重传 |
| **选了什么** | 最终方案 + 关键配置 | `MessageChatMemoryAdvisor` + `FileBasedChatMemory` |
| **后果是什么** | 好处、代价、后续风险 | 开发快；重启可恢复；长对话 token/文件变大 |

```text
问题 → 方案池 → 取舍 → 二阶效应（上线后会怎样）
```

## 二、TravelApp 方案对比表

### 2.1 多轮对话：如何实现「记忆」？

| 维度 | A. 无记忆 | B. 手写 ChatMemory | C. MessageChatMemoryAdvisor ✅ | D. PromptChatMemoryAdvisor |
|------|-----------|-------------------|-------------------------------|---------------------------|
| **做法** | 每轮只发当前 user 消息 | 业务代码里 get → 拼 Prompt → add | Advisor 自动读写 | 历史压进 System 文本 |
| **代码量** | 最少 | 多，易漏步骤 | 少，业务只传 id | 少 |
| **结构** | 无上下文 | 取决于你怎么拼 | 保留 user/assistant 角色 | 角色边界模糊 |
| **可扩展** | 差 | 改一处动全局 | 可叠 RAG、日志 Advisor | 与 RAG 混在 System 易乱 |
| **适用** | 单次问答 Demo | 理解原理时用 | **业务主线（本项目）** | 老 Completion API |
| **后果** | 无法「改第 2 天行程」 | 维护成本高、易 bug | 重启丢会话（纯内存时） | 长对话 System 膨胀 |

**本项目选择（2026-06-03 更新）**：C + `FileBasedChatMemory`（Kryo 写 `tmp/chat-memory/*.kryo`，重启可恢复）。  
**实验记录**：曾用 `MessageWindowChatMemory.maxMessages(2)` 验证窗口过小会失忆，后改为文件版（无条数上限，需注意 token）。

### 2.2 编排层：ChatModel 还是 ChatClient？

| 维度 | ChatModel | ChatClient ✅ |
|------|-----------|---------------|
| **抽象层级** | 底层，一次 `call(Prompt)` | 上层，Fluent + 默认 System + Advisor |
| **多轮** | 自己拼 Message 列表 | Advisor 自动注入历史 |
| **流式 / 结构化** | 需另接 API | `.stream()`、`.entity()` 统一入口 |
| **后果** | 第 02 章学习够用 | 第 03 章起业务必须用 |

### 2.3 System Prompt：写在哪？

| 维度 | Java 字符串 | `resources/prompts/*.st` ✅ | 百炼控制台 only |
|------|-------------|---------------------------|----------------|
| **迭代** | 改代码重编译 | 改文件即可 | 与代码脱节 |
| **协作** | 研发独占 | 产品/运营可改文案 | 难版本管理 |
| **后果** | 快但不利于调 Prompt | 与 Git 版本对齐 | 适合验证，不适合生产 |

### 2.4 会话 id：谁生成？

| 维度 | 后端 UUID 自动生成 ✅ | 前端必传 |
|------|----------------------|----------|
| **体验** | 首次调用即可用 | 需先拿 id 再续聊 |
| **风险** | 前端不传 id 则每轮新会话 | 前端丢失 id 则断上下文 |
| **后果** | Knife4j 调试友好；上线需约定 API 文档 | 更适合正式产品 |

**误用警示**：代码**不会**拦截空/错 `conversationId`，只会静默新建 UUID → 表现为「失忆」，排查时先查 id 是否一致。

### 2.5 结构化输出：手解析 JSON 还是 `.entity()`？

| 维度 | 手解析 `ObjectMapper` | `.entity(Record.class)` ✅ |
|------|----------------------|---------------------------|
| **做法** | 自己写 Prompt「返回 JSON」+ 解析 + 校验 | Spring AI 生成 Schema + 自动映射 |
| **类型安全** | 易漏字段、类型错 | `TravelReport(title, suggestions)` 编译期约束 |
| **后果** | 模型格式飘了要手写修复 | 模型不守 Schema 仍可能失败；需选好模型 |

**本项目**：`TravelReport` + `/travel/report`；报告 System 追加「XX的旅行报告」约束。

### 2.6 记忆持久化：内存 / 文件 / 数据库？

| 维度 | InMemory / Window | 文件 Kryo ✅ | MySQL / Redis |
|------|-------------------|-------------|---------------|
| **重启** | 丢失 | 保留 | 保留 |
| **多机** | 不行 | 不行（本地盘） | 可以 |
| **复杂度** | 最低 | 中（自定义 ChatMemory） | 高（需库） |
| **后果** | 适合 Demo | 适合单机学习/小产品 | 适合生产 |

**为什么 Kryo 不用 JSON 存 Message？** Message 是接口、子类多、无统一 JSON 结构；Kryo 序列化对象图更省事（课程选型）。

### 2.7 Prompt：拼接 / 读文件 / 模板填空？

| 维度 | Java `+` 拼接 | 读 `.st` 纯文本 | `SystemPromptTemplate` ✅ |
|------|--------------|----------------|--------------------------|
| **维护** | 最差 | 好 | 好 + 可填 `{today}` |
| **谁改文案** | 研发改 Java | 改文件 | 改文件 + Java 只传变量 |
| **是否必须** | ❌ | 够用 | 动态上下文时推荐 |
| **后果** | 难版本化 | 日期等写死在文件 | 模板与数据分离 |

**生活类比**：拼接 = 便利贴粘 Prompt；读文件 = 印好的手册；模板 = 手册留空每天填日期。

### 2.8 多模态：要不要在第 3 天接入？

| 维度 | 纯文本 chat | 多模态 VL |
|------|------------|-----------|
| **模型** | `qwen-plus` | `qwen-vl-plus` 等 |
| **输入** | `.user(text)` | `.user().text().media(图)` |
| **课程要求** | 主线必做 | **了解即可**，不挡进第 04 章 |
| **与 RAG** | — | RAG 检文档；多模态看用户传的图 |

## 三、费曼讲解：TravelApp + MessageChatMemoryAdvisor

> **标准**：能讲给没学过 Spring AI 的同学听，不用 jargon 堆叠。讲不顺 = 理解有洞。

### 3.1 用一句话说清

**TravelApp** 是旅行规划的「前台」：用户说一句话，它交给大模型，并把 **System 人设** 和 **历史聊天** 一起打包发送。  
**MessageChatMemoryAdvisor** 是「自动记账员」：调用模型前把本桌（`conversationId`）之前的对话翻出来，调用后再把本轮问答记上去。

### 3.2 生活例子：餐厅点菜

| 角色 | 对应代码 |
|------|----------|
| 你（顾客） | `message`：「把第 2 天改成西湖」 |
| 桌号 | `conversationId`：区分不同客人 |
| 服务员 | `MessageChatMemoryAdvisor` |
| 记事本 | `FileBasedChatMemory`（写在 `tmp/chat-memory/*.kryo`，换班也能翻） |
| 厨房规矩（厨师手册） | `SystemPromptTemplate(travel-system.st)` + `{today}`、`{userName}` |
| 厨房 | `ChatModel`（百炼 Qwen） |

没有 Advisor：每次换一个新服务员，不知道你之前点了「杭州 3 天」。  
有了 Advisor：服务员自动翻记事本，厨房才能理解「第 2 天」指行程里的第二天。

### 3.3 一次 `doChat` 里发生了什么（按时间顺序）

```text
1. 用户 POST /api/travel/chat，带 message + conversationId
2. TravelApp 若 id 为空则 new UUID
3. chatClient.prompt().user(message)
4. .advisors(a -> a.param(CONVERSATION_ID, chatId))  ← 告诉 Advisor 查哪本账
5. 【Advisor 前置】从 ChatMemory get(chatId) → 历史 Message 列表
6. 拼成：System + 历史 user/assistant + 本轮 user
7. ChatModel 调用百炼 API
8. 【Advisor 后置】add(chatId, 本轮 user + assistant) → 写入 .kryo
9. 返回 content 给 Controller
```

**结构化报告 `doChatWithTravelReport` 额外一步：**

```text
.system(渲染后 Prompt + 「生成旅行报告」约束)
→ .entity(TravelReport.class)  // 非 .content()
→ 得到 title + suggestions
```

### 3.4 自测：讲完若被追问这些，你能答吗？

| 追问 | 参考答案要点 |
|------|--------------|
| 和 `PromptChatMemoryAdvisor` 差在哪？ | Message 保留 role 结构；Prompt 把历史塞进 System 一大段字 |
| 重启应用历史还在吗？ | **在**，`FileBasedChatMemory` 读 `.kryo`；纯 InMemory 则不在 |
| 为什么曾实验 `maxMessages(2)`？ | 验证窗口过小会挤掉「杭州3天」；文件版无窗口要自控 token |
| `MyLoggerAdvisor` 会改回复吗？ | 不会，只观测；生产要注意脱敏 |
| Advisor 顺序谁决定？ | `getOrder()`，不是代码书写顺序 |
| 能不能不用 Advisor？ | 能，自己 get/add，但 RAG、日志也要手写 |
| `.entity()` 和 `.content()`？ | 前者映射 Java 对象；后者任意字符串 |

## 四、5 Whys 专题

### 4.1 为什么要用 Advisor？（多轮对话）

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 为什么 TravelApp 要接 Memory？ | 用户会分多轮补充预算、改行程 |
| 2 | 为什么不让 ChatModel 自己记？ | 大模型 API 无状态，每次请求独立 |
| 3 | 为什么用 Advisor 而不是在 doChat 里手写？ | 读写记忆是横切逻辑，应与业务解耦 |
| 4 | 为什么解耦重要？ | 后续加 RAG、日志、鉴权只加 Advisor，不改 doChat |
| 5 | **本质** | **责任链模式**：可插拔的「模型调用前后处理管道** |

### 4.2 为什么选 MessageChatMemoryAdvisor？

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 为什么不用 Prompt 方式塞历史？ | 现代 Chat API 认 message 数组 + role |
| 2 | 为什么 role 重要？ | 模型区分「用户说的」和「自己之前回的」 |
| 3 | 为什么课程推荐 Message？ | 与 GPT/Qwen 对话格式一致，上下文更稳 |
| 4 | 什么时候用 Prompt 方式？ | 极老模型或必须把历史写进 System 时 |
| 5 | **本质** | **数据结构匹配 API 契约，而不是图省事拼字符串** |

### 4.3 为什么业务主线是 Spring AI？

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 为什么学 SDK/HTTP 还要 Spring AI？ | 前两者理解协议；后者做产品 |
| 2 | 为什么产品要用框架？ | ChatClient、Advisor、RAG、MCP 同一套抽象 |
| 3 | 为什么换模型只改配置？ | 依赖 `ChatModel` 接口，不绑死百炼 |
| 4 | 代价是什么？ | BOM 版本要对齐，依赖冲突要会排查 |
| 5 | **本质** | **用框架买「扩展点」，用版本管理买「稳定性」** |

### 4.4 为什么 Knife4j 要升级 springdoc？（Boot 3.4）

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 为什么打开 doc.html 报 500？ | `/v3/api-docs` 生成失败 |
| 2 | 为什么 NoSuchMethodError？ | springdoc 2.3 调了已删除的 `ControllerAdviceBean` 构造 |
| 3 | 为什么 Boot 3.4 会断？ | Spring Framework 6.2 改了内部 API |
| 4 | 为什么排除旧依赖再引 2.8.9？ | Knife4j 4.4 自带旧 springdoc，需显式覆盖 |
| 5 | **本质** | **传递依赖是「版本契约」，框架升级必须审计依赖树** |

### 4.5 为什么用 FileBasedChatMemory 而不是只内存？

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 用户痛点是什么？ | 关掉应用后续聊，不想重复说目的地 |
| 2 | 为什么内存不够？ | JVM 重启堆清空 |
| 3 | 为什么选文件而不是 MySQL？ | 课程先学自定义 `ChatMemory`；无 DB 依赖 |
| 4 | 为什么用 Kryo？ | Message 子类多，JSON 手解析易错 |
| 5 | **本质** | **实现 ChatMemory 接口即可换存储；Advisor 无感** |

### 4.6 为什么用 `.entity(TravelReport.class)`？

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 为什么要结构化报告？ | 前端/业务要固定字段，不是解析 Markdown |
| 2 | 为什么不让前端解析 AI 长文？ | 格式不稳定、易变 |
| 3 | 为什么用 record？ | 不可变、字段清晰、配合 Schema |
| 4 | 和 `AssistantPromptTemplate` 关系？ | 都在约束「输出长什么样」 |
| 5 | **本质** | **用类型系统 + Schema 买「接口契约」** |

### 4.7 为什么阶段 E 用 SystemPromptTemplate？

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 读纯 `.st` 不够吗？ | 够；但 `{today}` 等动态值不好办 |
| 2 | 为什么不用 Java 拼接？ | 人维护难，职责混乱 |
| 3 | `render` vs `createMessage`？ | 前者要 String；后者要 Message 给 ChatClient |
| 4 | 是否必须？ | 否；业务可跑；工程上更规范 |
| 5 | **本质** | **文案结构在文件，变量值在代码** |

### 4.8 为什么 API Key 要放环境变量？

| 层 | 问题 | 答案 |
|----|------|------|
| 1 | 为什么 SpringAiAiInvoke 读不到 TestApiKey？ | Spring AI 读 `DASHSCOPE_API_KEY` 环境变量 |
| 2 | 为什么写进 Active profiles 不行？ | profile 是 `spring.profiles.active`，不是 env |
| 3 | 为什么不能写进 Git？ | 泄露 Key = 资金与安全事故 |
| 4 | 生产怎么管？ | 密钥管理服务 / CI 注入，不进仓库 |
| 5 | **本质** | **配置分「环境身份」与「应用配置」，Key 属于前者** |

### 日记 6 · Step6 记忆窗口实验（2026-06-10 实测）

| 项 | 内容 |
|----|------|

|----|------|
| **决定** | 在 IDEA / 对话中配置 DashScope Key |
| **假设** | 配进 Active profiles 等价 env |
| **反证** | InvalidApiKey；Key 曾出现在日志/聊天中 |
| **修正** | 仅 Environment variables；**轮换已暴露的 Key** |
| **下次** | `.gitignore` 与 `MyLoggerAdvisor` 生产脱敏 |

## 六、规律栏（可复用的经验）

从踩坑中提炼、可带到下一章的 **规律**：

| # | 规律 | 来源 | 下一步怎么用 |
|---|------|------|--------------|
| 1 | **大模型调用是「按量付费的外部 API」**，先确认账户与 Key，再 debug 代码 | Arrearage | 上线前加余额告警 |
| 2 | **框架能力来自 BOM 对齐**，混版本比写错代码更难查 | Spring AI M6 冲突 | 只通过 BOM 拉 Spring AI 系依赖 |
| 3 | **环境变量 ≠ Spring Profile**，密钥走 env，环境切换走 profile | IDEA 配错 | 团队文档写清 Run Configuration 截图 |
| 4 | **传递依赖随 Boot 升级会断**，文档/OpenAPI 类库要单独验证 | springdoc 2.3 | 升级 Boot 后 curl `/v3/api-docs` |
| 5 | **无状态模型 + 有状态业务 = Memory 层**，Advisor 是标准挂载点 | Ch03 架构 | RAG 也用 Advisor，别塞进 Controller |
| 6 | **Demo 的 `@Component` 会在启动时调 AI**，干扰调试就注释掉 | 启动打印问候语 | invoke / rag Demo 默认不自动跑 |
| 7 | **Prompt 质量 > 换模型**，旅行场景先迭代 System 再调参数 | 百炼控制台调试 | `travel-system.st` 版本化 `v1/v2` |
| 8 | **每个技术选择写清四步**，避免「抄完能跑但说不清为什么」 | 本手册 | 每章作业加一条 5 Whys |
| 9 | **记忆有窗口上限**，缩小 `maxMessages` 会导致「失忆」，不是 Advisor 坏了 | Step6 实验 | 控制历史条数 = 控制 token 与上下文深度 |
| 10 | **ChatMemory 可替换存储**，Advisor 只认接口；文件/Kryo/Redis 是实现细节 | FileBasedChatMemory | 第 04 章 RAG 也别塞进 Memory 实现 |
| 11 | **结构化输出用类型 + Schema 买契约**，别指望前端解析 AI 长 Markdown | TravelReport | PDF 导出可复用同一 record |
| 12 | **Prompt 三层递进**：拼接 < 外置文件 < 模板填空；不是二选一 | 阶段 E 问答 | `{var}` 是单花括号 |
| 13 | **多模态与 RAG 不同数据源**：图是用户输入；RAG 是文档检索 | 课件 6.5 | 第 04 章做 RAG，识图可后做 |
| 14 | **conversationId 误用表现为失忆**，代码不校验，排查先对 id | 多轮实测 | API 文档写清「续聊必带 id」 |
